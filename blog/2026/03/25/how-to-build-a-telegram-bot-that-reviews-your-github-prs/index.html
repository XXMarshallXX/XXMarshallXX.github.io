How to Build a Telegram Bot That Reviews Your GitHub PRs with OpenClaw

If you spend enough time shipping code, you eventually start asking the same question: why am I still opening five tabs just to see whether a pull request needs attention?

That was the starting point for this build. I wanted a faster loop between GitHub activity and the place where I already pay attention: Telegram. Not just notifications, either. I wanted a bot that could actually review pull requests, summarize changes, flag obvious issues, and make it easier to decide what deserves a deeper look.

This post walks through how to build a Telegram bot that reviews your GitHub PRs using OpenClaw, GitHub webhooks, and an AI review layer. It is not a polished enterprise architecture guide. It is a practical, builder-to-builder breakdown of how to connect the pieces, what matters, and what I learned while putting it together.

If you are a developer, indie maker, or AI enthusiast looking for a useful automation project, this one is a great sweet spot. It is small enough to build over a weekend, but powerful enough to become part of your daily workflow.

Why build a Telegram bot for GitHub PR reviews?

Most GitHub automation starts with notifications, but notifications alone are noisy. A raw “new pull request opened” message does not help much when you are juggling multiple repos, side projects, and production work. What you really want is context.

A Telegram bot gives you a simple interface for that context. Instead of opening GitHub and reading through each PR manually, you can receive a structured message with:

  • The PR title, author, and branch details
  • A summary of changed files
  • A quick AI-generated review of risky areas
  • Suggestions for follow-up questions
  • A direct link back to GitHub

Telegram is especially good for this because it is lightweight, fast, and easy to integrate. Messages feel immediate, and the bot can evolve from passive alerts into an interactive teammate. You can ask it to re-review a PR, summarize comments, or explain why a change might be risky.

That is where OpenClaw becomes interesting. It gives you a builder-friendly way to wire together GitHub events, LLM reasoning, and outbound messaging without reinventing every part of the stack.

The architecture: GitHub webhook in, Telegram review out

The core flow is straightforward:

  • GitHub sends a webhook when a pull request is opened, updated, or marked ready for review
  • Your backend receives the webhook and validates the signature
  • The service fetches PR metadata, changed files, and patch content from the GitHub API
  • OpenClaw orchestrates the review step with your chosen AI model
  • The result is formatted into a Telegram-friendly message
  • Your Telegram bot sends the review directly to you or your team chat

You can keep the first version extremely simple. One webhook endpoint, one review function, one Telegram message. The trick is not complexity. The trick is making the output useful enough that you trust it.

In my setup, I focused on three goals:

  • Keep the review concise enough to read on mobile
  • Surface real code concerns instead of generic praise
  • Preserve links back to the exact PR for deeper inspection

That meant the AI prompt mattered as much as the infrastructure. If your bot just says “looks good overall” on every pull request, nobody will keep using it.

Step 1: Create the Telegram bot and GitHub integration

Start with Telegram. Use BotFather to create a new bot and get your bot token. Once you have the token, send a test message from your backend to confirm the bot can reach your chosen chat or channel.

Next, set up a GitHub webhook on the repository you want to monitor. Point it to a public endpoint in your app, and subscribe at minimum to pull request events. In most cases, you will care about actions like:

  • opened
  • synchronize
  • reopened
  • ready_for_review

Make sure you verify the GitHub webhook signature. It is a small step, but it is one of those things that is easy to skip during a prototype and annoying to retrofit later.

At this point, your app should be able to receive a PR event and log useful fields like repository name, PR number, title, author, and URL. That is your first checkpoint. Before adding AI, make sure the plumbing works.

If you are using OpenClaw as the build foundation, this is where it helps to think in modules: one piece receives events, another fetches repository context, and another handles model execution. Keeping those responsibilities separate makes iteration much easier later.

Step 2: Fetch the pull request diff and prepare review context

Once the webhook arrives, you need more than the event payload. The interesting part is the code diff.

Use the GitHub API to fetch:

  • PR title and description
  • Changed files
  • Additions and deletions
  • Patch snippets where available
  • Review comments if you want follow-up analysis

This step looks simple, but it is where most practical constraints show up. Large pull requests can blow up token budgets quickly. Binary files and generated code are not useful review targets. Some patches are truncated. And if you dump the entire diff into a model every time, you will pay for it in latency and cost.

The better approach is to preprocess intelligently. Filter out lockfiles, generated assets, and vendored dependencies. Group files by type. Truncate enormous diffs. If needed, summarize file-level changes before asking the model for a final review.

I found it helpful to give the model a compact review packet with:

  • PR title and description
  • Repository name
  • List of changed files
  • Selected diff excerpts
  • Clear instructions on what to look for

This is the difference between “AI reads a blob of code” and “AI performs a code review task.” The more intentional the input, the better the output.

Step 3: Use OpenClaw to generate an AI PR review

This is the heart of the build. OpenClaw acts as the layer that ties the event-driven workflow to the model call and lets you shape the output into something actionable.

The prompt should be opinionated. Tell the model exactly what kind of review you want. For example, ask it to focus on:

  • Potential bugs or logic errors
  • Security concerns
  • Performance regressions
  • Missing tests or edge cases
  • Readability and maintainability issues

Also tell it what not to do. Do not ask for generic compliments. Do not invent context that is not present in the diff. Do not claim certainty when the change cannot be fully evaluated from the patch alone.

A useful pattern is to ask for a structured output like:

  • One-sentence summary
  • Risk level: low, medium, or high
  • Top 3 concerns
  • Suggested reviewer questions
  • Confidence note

This structure translates nicely into Telegram and makes the bot feel consistent. OpenClaw is especially useful here because it gives you a repeatable way to define and evolve the review workflow instead of scattering prompt logic across random scripts.

One behind-the-scenes lesson: the best AI review is often not the most verbose one. Developers scanning Telegram want signal, not a wall of text. Tight formatting wins.

Step 4: Send the review to Telegram in a way people will actually read

Formatting matters more than expected. Telegram is not a code review tool. It is a messaging app. So the bot message should feel native to that environment.

A good message template includes:

  • Repository and PR number
  • PR title
  • Author
  • AI summary
  • Risk level emoji or label
  • Bullet list of concerns
  • Link to open the PR in GitHub

For example, instead of dumping raw JSON or overlong markdown, send a compact message that can be read in under 20 seconds. If the model found no obvious concerns, say that clearly, but include a confidence caveat. If the change is risky, make that visible immediately.

You can also add commands so the bot becomes interactive. Useful options include:

  • /review 123 to re-run analysis on PR #123
  • /summary 123 to get a shorter version
  • /files 123 to list changed files
  • /comments 123 to summarize reviewer discussion

This is where the project starts to feel less like automation and more like a real developer tool.

Step 5: Add guardrails, caching, and workflow polish

Once the happy path works, the next layer is reliability. GitHub events can arrive often. Developers may push five commits to the same PR in ten minutes. If your bot reprocesses everything from scratch every time, the experience gets noisy fast.

A few practical improvements go a long way:

  • Cache recent PR analyses and only re-review changed files
  • Debounce rapid synchronize events
  • Skip draft pull requests until they are marked ready
  • Store previous reviews for comparison
  • Log model inputs and outputs for debugging

You should also think about security and privacy. If you are reviewing private repositories, make sure tokens are stored securely and prompts do not leak sensitive code into the wrong environment. Be deliberate about where model calls are executed and what data is retained.

Another useful upgrade is team routing. Instead of sending every review to one personal chat, route frontend PRs to one channel, backend PRs to another, or DM the code owner automatically. Small touches like that make the bot fit into real workflows.

What surprised me while building this

The biggest surprise was that the hard part was not GitHub or Telegram. It was deciding what kind of review is actually valuable in a chat interface.

There is a temptation to make the bot do everything: summarize, review, label, comment, assign, prioritize. But the more it tries to do, the easier it is for the signal to get diluted. The best version of this bot stayed narrow. It answered one core question well: what should I pay attention to in this pull request right now?

The second surprise was how much trust depends on restraint. If the bot confidently flags fake issues, people stop reading. If it admits uncertainty and points to specific risky changes, people lean in. That sounds obvious, but it changes how you design prompts, thresholds, and message formatting.

And finally, building this with OpenClaw made the iteration loop much faster. Instead of treating every change like a backend rewrite, I could evolve the workflow as a product: improve the prompt, adjust the event triggers, tighten the Telegram message, and test again.

Conclusion

If you want to build a Telegram bot that reviews your GitHub PRs, the stack is refreshingly approachable: GitHub webhooks for events, Telegram for delivery, and OpenClaw for the AI-powered orchestration in the middle.

The result is more than a notification bot. It becomes a lightweight review assistant that helps you triage changes, catch issues earlier, and stay closer to your codebase without constantly context-switching into GitHub.

For developers and makers, this is exactly the kind of build that feels useful immediately. It is practical, extensible, and a great example of how AI can augment real software workflows instead of just generating novelty.

If you build it, keep the first version focused. Get the webhook working. Fetch the diff. Generate a structured review. Send a clean Telegram message. Then iterate from there.
